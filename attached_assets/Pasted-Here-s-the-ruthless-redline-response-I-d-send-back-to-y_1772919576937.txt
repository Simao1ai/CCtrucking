### What I think is missing or understated

#### 1. We need a productization phase before or alongside branding cleanup

Right now the system is not just branded for CC Trucking — it also appears to contain **CC-specific workflows and assumptions** around services, compliance timelines, bookkeeping, preparer workflows, AI categorization, and portal behavior. For example, the platform already includes seeded trucking service items, recurring compliance templates, bookkeeping subscriptions, receipt scanning, preparer assignment, and trucking-specific AI categories.

We first need to classify what is:

* core platform logic
* tenant-configurable logic
* optional modules
* CC-specific custom behavior that should not live in the core product

Without that, we risk turning CC Trucking’s internal software into a multi-tenant version of the same internal software, instead of a true sellable SaaS product.

#### 2. Roles and permissions need a full redesign, not just a super-admin role

Today the platform uses roles like owner, admin, client, and preparer. 
Once we add SaaS tenancy, we likely need a more complete authorization model, such as:

* platform owner
* platform support/admin
* tenant owner
* tenant admin/manager
* tenant employee/staff
* client portal user
* preparer / reviewer
* read-only or auditor roles

This should be designed upfront, because once multi-tenant permissions get scattered through the codebase, it becomes expensive to unwind later.

#### 3. AI needs its own tenant-aware architecture

The current plan mentions dynamic prompts using the tenant company name, but that is not enough. The system already uses AI for tax document analysis, transaction categorization, and receipt extraction.  

We need explicit design for:

* tenant-specific AI prompts/instructions
* tenant-specific knowledge base/context
* tenant data isolation for AI workflows
* usage tracking and limits by tenant
* logging/audit trail for AI actions
* configurable vs locked AI behavior

AI is one of the easiest places for data leakage or inconsistent behavior if it is not tenant-scoped from the start.

#### 4. Billing is broader than Stripe subscriptions

Stripe integration is only the implementation detail. The real open questions are:

* what the pricing model is
* whether pricing is per company, per seat, per module, or usage-based
* what counts as billable usage
* what happens on failed payment
* how plan restrictions affect features
* whether onboarding is self-serve or white-glove

We should define the commercial model before building the billing layer.

#### 5. Onboarding is underspecified

A real onboarding flow for this platform likely includes more than signup and seeding. Based on the current feature set, onboarding may require:

* tenant branding setup
* service catalog setup
* workflow defaults
* user invitations
* role assignment
* bookkeeping/preparer setup
* form templates
* AI instructions or company knowledge
* document import / data migration

Given how much operational structure already exists, onboarding is likely a bigger workstream than currently described.

#### 6. Tenant-aware storage, notifications, jobs, and integrations need to be called out explicitly

The plan mentions general data isolation, but it should be more specific. This platform includes document storage, PDF invoices, signature requests, push notifications, scheduled reminders, recurring compliance ticket creation, and background jobs.  

That means multi-tenancy must include:

* file storage partitioning by tenant
* tenant-safe notification routing
* tenant-safe background job execution
* tenant-aware recurring schedulers
* tenant-aware API keys / external integrations
* tenant-aware email templates and sender identities

These are not side details. They are part of the core architecture.

#### 7. Support and internal platform tooling are missing

If this becomes a SaaS, we will likely need internal tools such as:

* support impersonation / safe admin assist mode
* tenant health/status view
* feature flags
* audit views by tenant
* usage dashboards
* rollout controls
* platform-wide logging and monitoring

This may not be phase 1, but it should be in the roadmap.

#### 8. Analytics should be split into platform analytics vs tenant analytics

The product already has owner dashboards, revenue metrics, SLA tracking, AR aging, and employee grading. 
In a SaaS version, we need two separate layers:

* analytics for each tenant
* analytics for the platform owner across all tenants

That separation should be intentional, not an afterthought.
